## 1. Executive Summary

Following repos explored:

| Component | Role | Key Pattern |
|-----------|------|-------------|
| **SciAgent-toolkit** | Infrastructure orchestrator | Roles activate agents + skills + MCP profiles via symlinks |
| **claude-scientific-skills** | 170+ reference skill library | SKILL.md + references/ + scripts/ + assets/ per skill |
| **life-sciences** | Plugin marketplace (18 tools) | MCP servers + local skills with waypoint/checkpoint patterns |
| **RNAseq-toolkit** | Reusable R/Python analysis code | source()-based modules, separation of compute/viz/IO |
| **DC_hum_verse .claude/** | Project-level config | Symlinks from toolkit, MCP servers, output style, addons |

### Architecture at a Glance

```
SciAgent-toolkit (git submodule)
    |
    |-- roles/*.yaml          # Declare: agents + skills + mcp_profile
    |-- agents/*.md           # 8 agent definitions
    |-- skills/*.md           # 40+ skill reference docs
    |-- templates/
    |   |-- mcp-profiles/     # Base MCP configs (overwritten on switch)
    |   |-- mcp-addons/       # Persistent MCP configs (merged on switch)
    |   |-- vendor/           # CLAUDE.md, AGENTS.md templates
    |   '-- gemini-profiles/  # Gemini CLI mappings
    |-- scripts/              # activate-role.sh, switch-mcp-profile.sh, etc.
    '-- docs/guidelines/      # Methodology (R/Python conventions)
         |
         v
Project .claude/ directory
    |-- agents/    -> symlinks to toolkit agents/
    |-- skills/    -> symlinks to toolkit skills/
    |-- settings.local.json   (permissions, MCP servers)
    |-- active-addons.json    (persistent addon state)
    '-- output-styles/        (custom interaction styles)
```

## 2. SciAgent-toolkit Architecture

### 2.1 Four-Tier System

```
Tier 1: Roles          (what to activate)
Tier 2: Agents/Skills  (what's available)
Tier 3: MCP Profiles   (what tools are connected)
Tier 4: Addons         (what persists across switches)
```

### 2.2 Roles

Roles are YAML files in `roles/` that declare the complete context for a workflow type.

**Structure:**
```yaml
name: scrna-atlas
description: scRNA-seq atlas integration & annotation
mcp_profile: coding

agents:
  - docs-librarian
  - bio-interpreter
  - insight-explorer
  - captions
  - doc-curator
  - code-reviewer
  - handoff

skills:
  - anndata
  - scanpy
  - scvi-basic
  - scvi-scanvi
  # ... (14 skills total for this role)
```

**Current roles (5):**

| Role | Agents | Skills | MCP Profile | Purpose |
|------|--------|--------|-------------|---------|
| `base` | 7 | 71 | coding | Default bioinformatics |
| `scrna-atlas` | 7 | 14 | coding | scRNA-seq integration & annotation |
| `multiome-grn` | 7 | 13 | coding | Multiome GRN inference |
| `planning` | — | — | — | Planning-focused |
| (others) | — | — | — | — |

**Activation flow:**
```
activate-role.sh <role> --project-dir <path>
    1. Parse roles/<role>.yaml
    2. Clear .claude/agents/ and .claude/skills/
    3. Create symlinks: toolkit/agents/<name>.md -> .claude/agents/<name>.md
    4. Create symlinks: toolkit/skills/<name>.md -> .claude/skills/<name>.md
    5. Call switch-mcp-profile.sh <mcp_profile>
```

### 2.3 Agents (8 canonical)

All agents are markdown files with YAML frontmatter.

| Agent | Purpose | Model | Fire-and-Forget? |

### 2.5 MCP Profiles

Profiles are JSON templates in `templates/mcp-profiles/` that define which MCP servers to connect.

**Current profiles (7):**
- `coding` — Sequential Thinking + Context7 + PAL
- `hybrid-research` — All coding tools + ToolUniverse
- `research-lite` — Minimal research tools
- Others for specific workflows

**Profile switching:** `switch-mcp-profile.sh <profile>`
- Overwrites `.mcp.json` with profile template
- Substitutes `${ENV_VAR}` placeholders
- Merges active addons on top
- Also writes `.gemini/settings.json` and `~/.codex/config.toml`

### 2.6 Addons (Layered Persistence)

Addons are MCP servers that persist across profile switches. This is the key differentiator from profile servers.

**How addons work:**
```
Profile switch → writes profile servers to .mcp.json
                → reads .claude/active-addons.json
                → merges enabled addon servers INTO .mcp.json
                → addon servers survive the next profile switch
```

**Addon template format** (`templates/mcp-addons/*.addon.json`):
```json
{
  "_meta": {
    "name": "notebook-tools",
    "description": "Jupyter notebook read/search/edit",
    "requires_setup": false,
    "context_tokens_estimate": "~2k",
    "tool_permissions": ["mcp__notebook-tools__nb_overview", "..."]
  },
  "mcpServers": {
    "notebook-tools": {
      "type": "stdio",
      "command": "uvx",
      "args": ["--from", "git+https://...", "notebook-tools-mcp"]
    }
  }
}
```

**Management:** `manage-addon.sh enable|disable|list|status <addon-name>`

**Currently active addon in DC_hum_verse:** `notebook-tools`

## 3. Reference Repo: claude-scientific-skills

**Location:** `01_modules/.ref/claude-scientific-skills/scientific-skills/`
**Scale:** 170+ skills covering data access, analysis, workflow, and output generation

### 3.1 Skill Directory Structure

Each skill is a directory (not a single file) with up to 4 components:

```
skill-name/
    SKILL.md              # Required: metadata + navigation + triggers
    references/           # Optional: detailed topic guides (*.md)
    scripts/              # Optional: executable Python/Bash utilities
    assets/               # Optional: templates, configs, examples
```

**Complexity tiers:**
- **Simple:** SKILL.md only (~30% of skills)
- **Standard:** SKILL.md + references/ + scripts/ (~30%)
- **Complex:** SKILL.md + references/ + scripts/ + assets/ (~40%)

### 3.2 SKILL.md Frontmatter

```yaml
---
name: skill-identifier         # kebab-case, matches directory name
description: "Trigger phrase and guidance"
license: MIT
metadata:
    skill-author: K-Dense Inc.
    website: https://...
    tags: ["bioinformatics", "single-cell"]
allowed-tools: Bash Read Write Edit    # Optional permission scoping
---
```

### 3.3 Cross-Skill Composition

Skills reference each other via:
- **Complementary Skills sections** — explicit "pairs well with" lists
- **Description cross-references** — "For X use skill-Y instead"
- **In-narrative mentions** — "See the `scanpy` skill for..."

**Conceptual layering:**
```
Data Access:     pubmed, ensembl, chembl, bioservices, gget
                    |
Analysis:        scanpy, anndata, scvi-tools, rdkit, matplotlib
                    |
Workflow:        literature-review, clinical-decision-support, EDA
                    |
Output:          scientific-writing, latex-posters, citation-management
```

### 3.4 Key Difference from SciAgent-toolkit Skills

| Aspect | claude-scientific-skills | SciAgent-toolkit |
|--------|------------------------|------------------|
| Format | Directory with SKILL.md + subdirs | Single .md file |
| Executable code | scripts/ directory with .py files | No executable code |
| Assets/templates | assets/ directory | None |
| Scale | 170+ skills | 40+ skills |
| Scope | General scientific | Bioinformatics-focused |
| Activation | Copy to ~/.claude/skills/ | Symlink via role activation |
| Cross-references | Complementary Skills sections | Implicit in skill text |

## 4. Reference Repo: life-sciences

**Location:** `01_modules/.ref/life-sciences/`
**Structure:** Claude Code Plugin Marketplace with 18 tools

### 4.1 Two-Tier Plugin Architecture

**Tier 1: Remote MCP Servers** (12 directories)
- Each has `.claude-plugin/plugin.json` manifest
- HTTP connections to external services (PubMed, BioRender, 10x Genomics, etc.)

**Tier 2: Local Skills** (6 directories)
- Self-contained workflows: SKILL.md + references/ + scripts/
- Notable: single-cell-rna-qc, scvi-tools, nextflow-development, clinical-trial-protocol

### 4.2 Marketplace Registry

Central `marketplace.json` with plugin metadata, tags, and skill paths.

### 4.3 Patterns Relevant to Reproducible Workflows

**Waypoint Pattern** (clinical-trial-protocol-skill):
- Saves intermediate results as JSON/Markdown in `waypoints/`
- Tracks completion status per step
- Enables resume from last completed step
- Prevents data loss on interruption

**Staged Execution with Validation Gates** (nextflow-development):
```
Step 0: Initialize
    | validation required
Step 1: Environment Check
    | MUST PASS before proceeding
Step 2-N: Execution (staged)
    | resume from waypoints
Final: Verification & Output
```

**Decision Tree Pattern** (scvi-tools):
```
Have scRNA-seq?
    Have labels? -> scANVI
    No labels?   -> scVI
Have CITE-seq?   -> totalVI
Have multiome?   -> MultiVI
```

**Progressive Complexity** (single-cell-rna-qc):
- Approach 1: One-command convenience script
- Approach 2: Modular building blocks for customization

## 5. RNAseq-toolkit: Code Module Patterns

**Location:** `01_modules/RNAseq-toolkit/` (git submodule, dev branch)
**Language:** R (primary), Python (GSEA plotting alternative)

### 5.1 Directory Organisation

```
scripts/
    General/          # Core utilities: annotation, DGEList, file I/O
    DE/               # Differential expression plots: volcano, PCA, MD
    GSEA/
        GSEA_processing/    # Analysis: run_gsea, normalize, pathway extraction
        GSEA_plotting/      # R visualization: dotplot, barplot, heatmap, running sum
        GSEA_plotting_python/  # Python alternative renderers
    ORA/              # Over-Representation Analysis
    custom_minimal_theme.R   # Publication ggplot2 theme
    utils_plotting.R         # DRY utilities: ensure_dir, save_plot, log_message
```

### 5.2 Code Style Conventions

- **Naming:** `snake_case` for functions and files
- **Indentation:** 2 spaces
- **Documentation:** Roxygen2 `#'` docstrings with `@param`, `@return`, `@export`
- **Sourcing:** Individual `source()` calls (not a package)
- **Load messages:** `message("[RNAseq-toolkit] Loaded: <function>")` on source

### 5.3 Design Principles

1. **Separation of concerns:** Processing never plots; plotting never computes
2. **Checkpointing:** Expensive computations cached via `saveRDS()`
3. **Decision by FDR, display by raw-p:** Statistical rigor in volcano plots
4. **Show all, highlight significant:** GSEA dotplots show top pathways, emphasise significant ones
5. **Single source of truth:** Colors, thresholds, schemas defined once
6. **Normalise then visualize:** In pipeline like scripts, or extractable modules functions never mix compute and visualisation. Separate them always, to be able to quickly iterate vis on cached compute results 

### 5.4 Sourcing Pattern in Analysis Projects

```r
# In DC_hum_verse analysis scripts:
source("01_modules/RNAseq-toolkit/scripts/GSEA/GSEA_processing/run_gsea.R")
source("01_modules/RNAseq-toolkit/scripts/GSEA/GSEA_plotting/gsea_dotplot.R")
source("01_modules/RNAseq-toolkit/scripts/custom_minimal_theme.R")
```

### 5.5 Function Tiers

| Tier | Purpose | Examples |
|------|---------|---------|
| Core wrappers | Wrap clusterProfiler/limma/edgeR | `run_gsea()`, `run_ora()` |
| Analysis pipelines | Multi-step automation | `run_gsea_analysis()`, `run_pooled_gsea()` |
| Data processing | Standardisation | `normalize_gsea_results()`, `get_pathway_genes()` |
| Visualization | ggplot2 plots | `gsea_dotplot()`, `create_standard_volcano()` |
| Utilities | Helpers | `ensure_dir()`, `save_plot()`, `format_pathway_name()` |

### 5.6 Testing

- Visual regression tests generating PDFs for manual inspection
- Edge case coverage (no significant genes, boundary cases, ggplot2 4.0+ compatibility)
- Test files: `tests/test_*.R`, output to `tests/output/`

## 7. Cross-Cutting Patterns

### 7.1 Pattern: Layered Configuration

All systems use some form of layered configuration where base settings can be extended without being overwritten:

| System | Base Layer | Extension Layer | Mechanism |
|--------|-----------|----------------|-----------|
| SciAgent-toolkit | MCP profiles | Addons | merge into .mcp.json |
| claude-scientific-skills | SKILL.md | references/ + scripts/ | directory structure |
| life-sciences | marketplace.json | plugin.json per tool | registry + manifest |
| RNAseq-toolkit | Function defaults | User parameters | R function arguments |

### 7.2 Pattern: Declarative Activation

Roles declare what should be active (YAML), scripts create the symlinks. The user never manually manages symlinks. This is the "single command activates everything" pattern:

```bash
activate-role.sh scrna-atlas --project-dir /workspaces/DC_hum_verse
```

### 7.3 Pattern: Separation of Concerns

Consistent across all components:
- **SciAgent-toolkit:** Roles vs Agents vs Skills vs Profiles vs Addons (5 concerns)
- **RNAseq-toolkit:** Processing vs Normalisation vs Visualisation vs IO (4 concerns)
- **claude-scientific-skills:** Navigation (SKILL.md) vs Knowledge (references/) vs Automation (scripts/) vs Templates (assets/) (4 concerns)

### 7.4 Pattern: Single Source of Truth via Symlinks/Submodules

```
Toolkit (canonical)  -->  Project (symlinks)  -->  Claude Code (reads)
```

Changes in toolkit propagate automatically. No copy-paste drift.

### 7.5 Pattern: Progressive Complexity

Every system offers simple entry points with paths to deeper customisation:
- **SciAgent-toolkit:** base role (everything) vs specialised roles (focused)
- **life-sciences:** Approach 1 (one-command) vs Approach 2 (modular)
- **RNAseq-toolkit:** `run_gsea_analysis()` (automated) vs `run_gsea()` + plotting (manual)

### 7.6 Pattern: Checkpoint/Waypoint Persistence

Multiple systems implement "save intermediate state, resume later":
- **life-sciences:** Waypoints in clinical-trial-protocol (JSON/Markdown)
- **RNAseq-toolkit:** `saveRDS()` checkpointing for expensive GSEA
- **DC_hum_verse:** `03_results/checkpoints/` for analysis milestones

## 8. Gap Analysis

### 8.1 What Exists and Works Well

- Role-based activation of agents + skills + MCP profiles
- Addon persistence across profile switches
- Symlink-based single source of truth
- Comprehensive skill library (40+ in toolkit, 170+ in reference)
- Modular R toolkit with clear separation of concerns
- Custom output style for pedagogical interaction

### 8.2 What's Missing or Underutilised

| Gap | Current State | Impact |
|-----|---------------|--------|
| **No phase-based workflow skills** | Skills are tool-reference docs, not workflow guides | No structured "explore -> integrate -> annotate -> analyse" progression |
| **No verification/checkpoint skills** | Verification is ad-hoc in notebooks | No systematic "check before proceeding" enforcement |
| **Skills are flat .md files** | SciAgent-toolkit skills lack references/, scripts/, assets/ subdirs | Less rich than claude-scientific-skills or life-sciences patterns |
| **No decision-tree routing** | User must know which skill to invoke | Unlike life-sciences scvi-tools decision tree pattern |
| **No waypoint/resume mechanism** | Sessions start fresh each time | No structured pickup from where you left off |
| **RNAseq-toolkit is R-only** | Python analysis (scanpy/scVI) uses ad-hoc code | No equivalent Python module library with same conventions |
| **Output style not role-aware** | cs101 style is manually set, not tied to roles | Different workflows might benefit from different interaction styles |
| **No skill for iterative data exploration** | EDA is unstructured | The user's described workflow (explore independently, then integrate) has no supporting skill |
| **Addons only cover MCP servers** | No addon pattern for skills or output styles | Can't persistently layer skills across role switches |

### 8.3 Opportunities from Reference Repos

From **claude-scientific-skills:**
- Adopt directory-based skill format (SKILL.md + references/ + scripts/) for richer skills
- Add "Complementary Skills" sections for discoverability
- Add "When NOT to Use" contraindications

From **life-sciences:**
- Adopt waypoint pattern for multi-session workflows
- Adopt validation gate pattern (must pass step N before step N+1)
- Adopt decision tree pattern for tool selection
- Adopt progressive complexity (quick vs modular approaches)

From **RNAseq-toolkit:**
- Apply same separation-of-concerns discipline to Python analysis code
- Apply same checkpoint pattern to scVI model training
- Apply same "show all, highlight significant" visual philosophy to scRNA-seq plots
